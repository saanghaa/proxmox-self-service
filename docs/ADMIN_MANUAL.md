# Proxmox Horizon 관리자 매뉴얼

이 문서는 관리자 계정이 Proxmox Horizon을 운영할 때 필요한 기능과 운영 기준을 설명합니다.

## 1. 관리자 운영 설계 철학

관리자 화면은 단순한 설정 패널이 아니라 운영 콘솔입니다.  
따라서 다음 원칙을 따릅니다.

- 자동화는 적극적으로 사용한다.
  - 반복적인 VM 배포, 상태 조회, 키 교체, 동기화는 시스템이 수행한다.
- 결정 권한은 관리자에게 남긴다.
  - 승인, 반려, 정책 변경, 토큰 확인, 운영 예외 처리는 관리자가 결정한다.
- 민감 기능은 재인증한다.
  - 토큰 보기, 키 다운로드 같은 기능은 OTP를 다시 확인한다.
- 운영 흔적은 남긴다.
  - 감사로그와 알림을 통해 “누가 무엇을 했는지” 추적 가능해야 한다.
- 완벽한 환경을 가정하지 않는다.
  - 여러 네트워크 대역, 자체서명 인증서, 수동으로 준비된 Cloud Image 같은 현실을 전제로 설계한다.

## 2. 관리자 화면 개요

주요 관리자 화면:
- 관리자 메인: `app/src/views/admin.ejs`

주요 기능 범주:
- VM 생성 요청 처리
- Proxmox 서버 연결
- 사용자 / 사용자 그룹 / 할당량 관리
- 보안 정책 관리
- 알림 설정
- 메뉴 / UI 텍스트 설정
- 감사로그 및 통계
- 백업과 복구

## 3. 관리자 로그인과 보안

관리자도 일반 사용자와 동일하게 로그인 후 OTP를 완료해야 보호 기능에 접근할 수 있습니다.

추가 OTP 재인증이 필요한 작업:
- Proxmox Token Secret 보기
- SSH 키 다운로드
- SSH 키 교체

운영 권장:
- 관리자 계정은 개인 계정으로 분리해서 사용
- 공용 계정 사용 최소화
- OTP 재설정은 감사로그 확인 후 수행

## 4. VM 생성 요청 처리

관리자는 사용자가 올린 VM 요청을 검토하고 승인, 반려, 배포합니다.

검토 항목:
- 요청 그룹
- 인스턴스 타입
- OS 선택값
- 수량
- 추가 디스크
- 요청 사유

승인 시 결정 항목:
- Proxmox 노드
- Cloud Image
- 네트워크 관련 값
- 필요 시 예외 조정

운영 원칙:
- 사용자가 선택한 값은 힌트이지 최종 실행값이 아닙니다.
- 실제 배포 가능한 리소스와 운영 정책이 우선합니다.

## 5. Proxmox 서버 연결

관련 기능:
- 노드 등록
- 수정
- 삭제
- 연결 테스트
- 토큰 보기

관련 코드:
- `app/src/routes/api/deploy.ts`
- `app/src/services/proxmox.ts`

### 5.1 등록 원칙

Proxmox Horizon은 기준 노드 하나를 입력받아 클러스터를 탐지할 수 있습니다.

현재 동작 원칙:
- 처음 입력한 노드는 사용자가 넣은 관리 주소를 그대로 사용
- 같은 클러스터의 다른 노드는 관리 네트워크 인터페이스를 탐색해서 자동 등록
- 비로컬 노드는 gateway가 있는 인터페이스를 우선 채택
- corosync 전용 주소 대역은 자동 관리 주소로 채택하지 않음

이유:
- 다중 대역 환경에서 잘못된 내부 전용 주소가 등록되는 문제를 줄이기 위해서입니다.

### 5.2 토큰 보기

토큰 보기 기능은 민감 정보 보호를 위해 OTP 재인증 후 일정 시간 동안만 Token Secret을 보여줍니다.

운영 원칙:
- Host와 Token ID는 목록에 이미 보이는 정보이므로 모달에서는 Token Secret 중심으로 최소 노출
- 토큰 조회 성공과 실패는 감사로그에 남김

### 5.3 삭제 시 주의

노드 삭제는 배포 대상과 상태 동기화 범위에 영향을 줍니다.

권장 절차:
1. 현재 노드가 실제 운영 대상인지 확인
2. 잘못 등록된 클러스터 노드인지 확인
3. 필요한 경우 기준 노드 하나만 남기고 재등록

## 6. Cloud Image와 Proxmox 호스트 준비

앱은 Cloud Image를 임의로 다운로드해 배포하지 않습니다.  
관리자가 사전에 검증한 이미지와 스니펫을 사용한다는 원칙을 따릅니다.

관련 스크립트:
- `scripts/proxmox-enable-content.sh`
- `scripts/proxmox-download-cloud-image.sh`
- `scripts/proxmox-token-rotate.sh`

권장 초기 순서:
1. 스토리지에 `import`, `snippets` 활성화
2. Cloud Image 다운로드
3. API Token 생성
4. 관리자 화면에서 노드 등록

## 7. 사용자, 그룹, 쿼터 관리

관리자는 다음을 제어할 수 있습니다.

- 사용자 생성 / 삭제 / 활성화 / 비활성화
- 관리자 권한 부여 / 해제
- 비밀번호 초기화
- OTP 재설정
- 그룹 생성 / 수정 / 삭제
- 그룹 멤버십 관리
- 그룹 쿼터 설정

운영 원칙:
- 사용자 계정 정책과 인프라 정책을 분리해 관리
- 그룹을 기준으로 권한과 자원 할당을 묶어 운영
- 예외 처리보다 표준 그룹 정책을 우선

## 8. 비밀번호 정책과 보안 정책

관리자는 시스템 전체 비밀번호 정책을 설정할 수 있습니다.

예:
- 최소 길이
- 대문자 / 소문자 / 숫자 / 특수문자 요구
- 만료 주기
- 경고 일수

또한 자동 키 교체 정책도 관리합니다.

자동 키 교체 정책 의도:
- 오래된 자격증명을 방치하지 않도록 함
- 교체 실패 시 다음 주기에 재시도 가능하게 함
- 관리자가 직접 모든 VM을 손으로 점검하지 않아도 되게 함

## 9. VM 관리

관리자는 사용자보다 더 넓은 운영 기능을 수행할 수 있습니다.

가능한 작업:
- Soft Delete된 VM 복원
- Proxmox와 DB에서 영구 삭제
- 수동 VM 등록
- 그룹 변경
- Job 연결

운영 원칙:
- 사용자 실수는 복구 가능한 방향을 우선
- 영구 삭제는 마지막 단계에서만 수행
- 외부에서 이미 존재하는 VM도 운영 목록에 편입 가능하게 함

## 10. VM 동기화와 마이그레이션 반영

관련 서비스:
- `app/src/services/vmSyncService.ts`

역할:
- 등록된 Proxmox 노드를 순회
- VM 상태와 디스크 구성, 노드 정보를 DB에 반영
- 수동 변경과 마이그레이션 결과를 운영 화면에 반영

운영 철학:
- DB는 운영 편의를 위한 조회 모델이지만 최종 소스는 Proxmox
- 따라서 일정 주기로 다시 맞추는 구조를 사용

## 11. 통계와 운영 가시성

관리자 화면의 통계 탭은 단순 리포트가 아니라 운영 판단 도구입니다.

대표 영역:
- Proxmox 노드 리소스 현황
- VM 배치 현황
- VM 분포 분석
- 요청/승인 현황
- 사용 패턴 분석

목적:
- 리소스 병목 확인
- 배포 흐름 병목 확인
- 사용자 활동 패턴 확인
- 그룹별 자원 편중 확인

## 12. 알림 설정

지원 채널:
- Slack
- Teams
- Custom Webhook

운영 원칙:
- 모든 이벤트를 다 보내는 것보다, 중요한 이벤트를 빠르게 파악하는 것이 우선
- 신규 채널은 핵심 보안/운영 이벤트 중심으로 시작

권장 우선 이벤트:
- 배포 실패
- 로그인 실패
- 키 다운로드
- SSH 키 교체
- 사용자 생성 / 삭제
- VM 삭제

## 13. 감사로그

감사로그는 운영 기록의 기준선입니다.

확인 가능한 대표 이벤트:
- 로그인 성공 / 실패
- OTP 관련 이벤트
- 사용자 생성 / 삭제
- VM 삭제 / 복원
- 배포 완료 / 부분 완료 / 실패
- 토큰 조회
- 키 다운로드 / 키 교체

운영 원칙:
- 알림은 실시간 대응용
- 감사로그는 사후 추적용
- 둘은 서로 대체 관계가 아니라 보완 관계

## 14. 백업과 복구

자세한 절차는 [백업 매뉴얼](./BACKUP_MANUAL.md)을 참고합니다.

관리자 관점 핵심 원칙:
- 백업은 복구 가능성을 보장할 때만 의미가 있음
- 정책 변경 전과 대규모 데이터 작업 전에는 백업 권장
- 암호화 키와 `.env` 관리도 복구 전략의 일부

## 15. 운영 중 자주 보는 문제

### 15.1 Cloud Image가 승인 화면에서 보이지 않음

점검:
- 스토리지에 `import` content가 활성화되어 있는지
- 이미지가 올바른 스토리지에 존재하는지
- 파일 형식과 볼륨 이름이 예상 패턴과 맞는지

### 15.2 스니펫 관련 배포 실패

점검:
- `snippets` content 활성화 여부
- `proxmox-cloud-init.yaml` 존재 여부
- 지정한 storage와 snippet 경로 일치 여부

### 15.3 노드가 오프라인으로 보이거나 잘못된 주소가 등록됨

점검:
- 잘못된 관리망이 자동 등록됐는지
- 기준 노드 입력값이 올바른지
- 다중 대역 환경에서 gateway가 있는 인터페이스가 관리망으로 잡히는지

### 15.4 토큰 보기나 민감 다운로드가 안 됨

점검:
- 관리자 계정 OTP 활성화 여부
- OTP 코드 오입력 여부
- 세션 만료 여부
